# Core Concepts

## Collections

A collection is a named vector index. It is the equivalent of an
index in Pinecone or a collection in Qdrant.

Each collection has a fixed vector dimension. Vectors upserted to
a collection must match that dimension.

## Write vs Query Latency

These are measured independently and should not be confused.

**Query latency** - reads from the in-memory HNSW index only.
No network calls. This is the 4.7ms p99 figure.

**Write latency** - `.upsert()` returns after the in-memory
insert (~2ms). Persistence to Shadow Drive and the Solana
Merkle root post happen asynchronously in the background.
Your application is never blocked waiting for either.

**Verify latency** - `.verify()` fetches the current Merkle
root from Solana and compares it locally. Expect ~400ms
due to Solana RPC round-trip. This is an optional integrity
check, not part of the normal read/write path.

## Merkle Proofs

After every upsert, VecLabs computes a SHA-256 Merkle tree over
all vector IDs and their values. The 32-byte root is posted to the
VecLabs Anchor program on Solana.

The root is deterministic - the same collection state always
produces the same root. Anyone with the collection data can
independently verify the root matches what is on-chain.

## Verification

Calling `index.verify()` does three things:

1. Computes the current Merkle root locally
2. Posts it to the Solana program via a transaction
3. Returns the transaction signature and Explorer URL

The transaction costs approximately $0.00025 (one lamport).
On devnet it is free.

## Wallets

VecLabs automatically generates and manages a Solana wallet
on your behalf. You never need to install a wallet extension,
fund an account, or interact with Solana directly.

The wallet serves two purposes under the hood:

- **Signing** - after every write, VecLabs signs and posts a
  32-byte Merkle root to Solana on your behalf. One transaction,
  $0.00025, 400ms finality.
- **Encryption** - your vectors are encrypted with AES-256-GCM
  using a key derived from your wallet before leaving the SDK.
  VecLabs cannot read your data.

Your wallet keypair is stored locally in your project and never
leaves your machine. You own it. If you already have a Solana
wallet you want to use instead, you can pass it explicitly:

```typescript
const sv = new SolVec({
  wallet: "/path/to/keypair.json", // optional - auto-generated if omitted
});
```

On devnet you can use any keypair. On mainnet you will use your
actual wallet. VecLabs never has access to your private key.
